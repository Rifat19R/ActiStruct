# ActiStruct v0.7.2 Release Notes

## QE-Free Dry-Run Candidate Selector for Future Live Validation

## 1. Summary

ActiStruct v0.7.2 adds a QE-free dry-run candidate selector that produces
schema-valid, review-only candidate rows for future live QE/PBE validation
planning. It does not run DFT, does not reuse historical completed records
as future candidates, and does not fabricate prediction or acquisition
values.

## 2. What Changed

- `analysis/dry_run_live_candidate_selector_v072.py` — a new, self-contained
  script that imports existing `generated_models/*.py` workflow definitions
  (safe to import — they only build configuration objects; QE launches only
  inside an `if __name__ == "__main__":` guard) and writes placeholder
  candidate rows for human review.
- `data/dry_run_live_candidates_v072.csv` — the generated output table.
- `reports/dry_run_live_candidate_selector_v072.md` — the design rationale,
  field-by-field computed-vs-pending breakdown, and safe/unsafe claim lists.
- `tests/test_dry_run_live_candidate_selector_v072.py` — 7 tests covering
  safe execution, schema completeness, dry-run status, and exclusion of
  historical-outcome columns.
- `docs/model_and_tests.md` — updated to document the new test file and the
  current test count.
- `README.md` — a short pointer to the new release note and the
  surrounding v0.7.x design/audit documents.

## 3. Main Output

`data/dry_run_live_candidates_v072.csv` contains:

```text
9 dry-run rows
5 material/workflow families (bulk_al, mos2, bulk_mgo_generated, bulk_si_generated, graphene_generated)
5 v0.7 design categories (exploitation, uncertainty_exploration, low_failure_risk, failure_risk_challenge, diversity)
```

Every row has `status = dry_run_only` and a `notes` field stating it is not
selected for QE/PBE execution and requires human review. Every science
field (`predicted_value`, `uncertainty_lcb_score`, `failure_risk`,
`acquisition_score`, `expected_runtime_risk`) is the literal string
`not_computed`.

## 4. Conservative Design Choices

- Candidate design-variable values are arithmetic placeholders within each
  workflow's documented `Variable.lo`/`Variable.hi` range, and the script
  programmatically checks (and raises an error on collision) that no
  placeholder value coincides with that workflow's documented
  `Variable.initial` seed values — guarding against accidentally presenting
  an already-evaluated point as an unresolved candidate.
- The real GP/LCB proposal path (`qe_active_inverse_common.py::_propose_inverse`)
  was deliberately not used here: it requires training on real
  `(params, energy)` pairs, which exist in this environment only as local,
  gitignored `outputs/cache/*.pkl` files (zero of which are committed to
  the repository) — using them would make the script non-portable, and
  fabricating energies to bootstrap a GP would be fabricating a scientific
  result. This release chose portability and honesty over a more elaborate
  but non-reproducible or fabricated approach.
- `selection_category` reflects only design *intent* for a future scored
  selector, not an actual acquisition score.

## 5. What This Release Does Not Do

ActiStruct v0.7.2 does not:

```text
run QE/PBE
launch pw.x or mpirun
create executable QE input files
reuse historical completed records as future candidates
fabricate prediction, uncertainty, failure-risk, or acquisition values
claim live DFT savings
claim new materials discovery
claim QE/PBE validation
claim that failure-aware ranking always outperforms LCB-only
```

## 6. Scientific Meaning

This release does not add new scientific evidence — it adds tooling and a
review artifact that prepares the ground for the live validation step
described in `reports/live_qe_validation_batch_design_v070.md`. The
governing principle remains unchanged:

```text
ML predicts. Uncertainty ranks. Failure risk warns. QE/PBE validates.
```

## 7. Tests

```text
pytest -q -> 81 passed
```

(74 tests prior to v0.7.2, plus 7 new tests in
`tests/test_dry_run_live_candidate_selector_v072.py`.) No QE/DFT is
launched by any test in the suite.

## 8. Why This Matters

The v0.7.1 candidate-source audit
(`reports/live_candidate_source_audit_v071.md`) found that no standing,
unresolved future-candidate pool existed anywhere in the repository — every
candidate-like file was either a historical completed record or a design
schema with zero rows. v0.7.2 closes that specific gap conservatively: it
produces a concrete, reviewable table of placeholder candidates without
crossing into fabrication or reusing historical outcomes, so a human
reviewer has something real to look at before any live batch is designed
further.

## 9. Current Status After v0.7.2

```text
branch: main

```

A schema-valid, review-only dry-run candidate table exists. No real

## 10. Remaining Limitation

There is still no portable, non-fabricated way to obtain real
`(params, energy)` training pairs for the GP surrogate without either
depending on local, gitignored QE caches or running new QE/DFT. Until that
is resolved, any future "real" (scored, not placeholder) candidate
selection step will face the same portability-vs-fabrication tradeoff
this release deliberately avoided.

## 11. Recommended Next Step

Resolve Section 10's limitation first — for example, by committing a small,
explicitly-labeled seed dataset of real `(params, energy)` pairs, or by
accepting that the first live batch trains on whatever local cache exists
on the machine that runs it. Only after that should a future task attempt
to produce *scored* (not placeholder) candidate rows, still without running
any new QE/DFT for the scoring step itself.
